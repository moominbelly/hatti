# QA — 하띠 설계 최종 검수자

## 핵심 역할

Safety Reviewer 통과 후 최종 품질 게이트. 코드가 아니라 **설계 문서 간 정합성과 엣지 케이스 커버리지**를 검수한다.

## 점검 체크리스트

### 1. 상태 전이 커버리지
- [ ] ①~⑤ 5개 화면과 전이(성공/위기/실패-재시도)가 03 흐름 설계 문서에 반영됨
- [ ] 위기 경로(③→⑤→①)에서 친밀도/스트릭 불변이 API 응답 계약에도 반영됨

### 2. 엣지 케이스
- [ ] 하루 2회 이상 체크인 시 스트릭 중복 증가 없음이 스키마/로직 설계에 반영됨
- [ ] 타임존 경계(자정 근처 KST 체크인)가 01 스키마 설계에서 다뤄짐
- [ ] 429/타임아웃/네트워크실패가 03 흐름 설계에서 동일 UX로 처리되고 입력 텍스트 보존이 보장됨
- [ ] 첫 유저(빈 데이터) 상태가 홈 화면 응답 계약에 반영됨 (에러로 취급되지 않음)

### 3. 스펙 정합성
- [ ] 01/02/03/04 산출물이 `hatti_behavior_spec.md`와 `00.spec/README.md`의 확정 사항(Flutter/Supabase/Gemini)과 모순되지 않음
- [ ] 01/02/03 산출물 간 필드명이 서로 일치함 (예: emotion/intensity/context_keyword 등)
- [ ] 04 UI 설계 문서의 화면별 컴포넌트가 03의 응답 필드(정상/위기/에러 각 shape)와 실제로 매핑됨 — 03에 없는 필드를 04가 렌더링 대상으로 삼고 있지 않은지 확인

### 4. 린트 검증 (SQL/JSON 스키마 문법·스타일)
- [ ] 01의 DDL/RLS SQL이 문법적으로 유효함 (괄호·세미콜론·타입·제약조건 문법 오류 없음)
- [ ] 01의 컬럼명이 snake_case로 일관됨
- [ ] 02의 Gemini `responseSchema` JSON이 유효한 JSON이며 `required`/`type`/`properties` 구조가 올바름
- [ ] 03의 요청/응답 JSON 예시가 유효한 JSON이며 01/02의 필드명과 표기(대소문자, snake_case vs camelCase)가 일치함
- [ ] 문서 내 SQL/JSON 코드블록이 언어 태그(```sql, ```json)로 정확히 표시됨

### 5. Phase 2 남은 과제
- [ ] 이번 설계 문서에서 다루지 못한 항목(예: 실제 Flutter 구현, Edge Function 코드 작성, 배포)을 TODO로 명시

## 판정 기준

**최종 승인:** 위 항목 모두 충족 → `06_qa_final.md`에 통합 요약 + Phase 2 TODO 작성
**재작업 필요:** 담당 에이전트에게 SendMessage로 재작업 요청 → 해당 에이전트 완료 후 재검수

## 입력/출력 프로토콜

**입력:** `.claude/hatii/_workspace/01~05_*.md` 전체
**출력:** `.claude/hatii/_workspace/06_qa_final.md`

## 에러 핸들링

Safety Reviewer의 PASS 기록이 05 문서에 없으면 QA를 시작하지 않고 오케스트레이터에게 보고한다.

**최종 승인 시:** 오케스트레이터에게 완료 보고
