# 하띠(Hatti) 도메인 분석 — 하네스 설계를 위한 리서치 기록

> 소스: `hatti_behavior_spec.md` (유일한 프로젝트 문서, 코드/스키마 파일 없음)
> 목적: 이 스펙을 구현하는 하네스(에이전트 팀)를 설계하기 위한 심층 분석

---

## 1. 제품 정체성

**하띠**는 감정 체크인 앱이다. 표면적으로는 "무드 트래커 + 반려 캐릭터" 형태지만, 스펙 전체를 관통하는 설계 철학이 명확히 존재한다:

| 축 | 하띠의 입장 | 경쟁 카테고리와의 차이 |
|---|---|---|
| AI의 역할 | **오직 공감(Empathy only)** | 무드 트래커(분류만) / AI 챗봇(교정·조언 시도)과 차별화 |
| 게임화 | **보상만, 패널티 없음** | 친밀도 비감소, 스트릭도 "유지"는 있어도 벌점은 리셋뿐 |
| 안전 | **위기 시 인간에게 연결, AI가 대신 상담하지 않음** | 공감 원칙의 유일한 예외 지점 |
| 데이터 | **감정 원문 = 민감 개인정보**, 삭제권 명시 | RLS 강제, 개인정보 처리 원칙 내재 |

이 철학이 코드 레벨 요구사항으로 그대로 번역된다 — 예를 들어 "공감만 한다"는 원칙은 실제로 Gemini 프롬프트에 대한 **금지어 리스트**(5장)로 구체화되어 있고, "위기 시 예외"는 상태 머신 분기(⑤)로 구체화되어 있다. 즉 이 스펙은 일반적인 PRD가 아니라 **거의 의사코드에 가까운 상태 정의서**다.

## 2. 시스템 아키텍처 (스펙에서 역산)

```
[React 프론트엔드]
   ├─ 화면: 홈 / 입력 / 분석중 / 응답 / 위기  (5-state 머신)
   │
[Supabase]
   ├─ Auth: 익명 로그인 (회원가입 없음)
   ├─ DB: hatti_state (친밀도/스트릭/단계)
   │      checkin_log (raw_text, emotion, intensity, context_keyword, empathy, affirmation, diary)
   ├─ RLS: 본인 행만 조회/삭제 가능
   └─ 시간대 판정: KST(UTC+9) 기준 — UTC 서버 시각과의 변환 필요 지점 (버그 위험 명시됨)
   │
[Gemini AI 레이어] — "1차+2차 호출"이라 명시됨 (2단계 파이프라인 추정)
   ├─ 1차: 감정 분류(라벨/강도/맥락태그) + 위기 신호 탐지(crisis_flag)
   ├─ 2차: 공감 대사 + 확언 생성 (+ "기억" 참조 조건부 반영) + 비동기 일기 생성
   └─ 안전필터: Gemini 자체 세이프티 블록도 위기 분기로 편입 (3중 방어의 3번째 축)
```

**핵심 리스크 지점 (스펙이 명시적으로 경고한 부분):**
1. **시간대 계산** — "서버는 UTC로 도므로 밤 체크인이 잘못 끊길 수 있음" → 스트릭 계산 로직에 타임존 버그 가능성을 스펙 저자가 이미 인지하고 경고
2. **위기 탐지 오탐/미탐** — "우울해", "너무 힘들어"가 위기로 오판되지 않아야 함(과탐) / 진짜 위기는 놓치면 안 됨(미탐). 3중 방어(키워드 프리필터 OR AI crisis_flag OR AI 세이프티 차단)로 재현율을 우선하는 설계
3. **타임아웃/쿼터 처리** — 10초 타임아웃, 429 처리 시 입력 텍스트 보존 필수 (재입력 강요 금지)
4. **"기억" 기능의 톤** — 반영이지만 평가가 아니어야 함 ("지난번에도 회의가 너를 데려갔었지" O / "또 회의 때문에 지쳤네" X) — 이건 프롬프트 엔지니어링 품질 문제이지 로직 문제가 아님

## 3. 상태 머신 요약

```
S0(익명로그인, 1회) → S1(홈) --[체크인시작]--> S2(입력) --[전송]--> S3(분석중)
S3 --[성공]--> S4(응답) --[마치기]--> S1 (친밀도+1, 스트릭 갱신)
S3 --[위기감지]--> S5(위기) --[홈으로]--> S1 (친밀도/스트릭 불변)
S3 --[실패/타임아웃/429]--> S3 (재시도 버튼, 입력 보존)
```

파생 상태 규칙:
- **친밀도**: monotonic increasing, per-checkin +1
- **단계**: 친밀도 구간 함수 (0-2/3-6/7+) → 캐릭터 비주얼 + 프롬프트 주입 톤 변경
- **스트릭**: `today_checked ? keep : (yesterday_checked ? +1 : reset_to_1)`, KST 기준
- **마일스톤**: 스트릭 3/7/14일 트리거 (보상 해금, 삭제 없음)

## 4. 이 도메인이 요구하는 전문 영역 (하네스 에이전트 후보)

기존 광고 어댑터 하네스(architect→engineer→test/security 병렬→qa)와 성격이 다르다. 이 프로젝트는:
- **백엔드/스키마**보다 **AI 페르소나 품질**과 **안전성**이 성패를 가른다
- 코드 정확성 검증(QA)만큼 **톤/카피 준수 검증**이 별도 축으로 필요하다 (일반 QA가 커버 못 함)
- **위기 대응 로직**은 보안 취약점이 아니라 **안전(safety) 취약점** — 별도 게이트 필요

식별되는 전문 영역:
1. **Supabase 스키마/RLS 설계** — hatti_state, checkin_log 테이블, RLS 정책, 스트릭 계산(KST 처리)
2. **Gemini 프롬프트 엔지니어링** — 1차(분류+위기탐지) / 2차(공감+확언+기억반영) 프롬프트 설계, 금지어 규칙 내재화
3. **React 프론트엔드 구현** — 5-state 화면, 로딩/에러/오프라인 처리(홈은 에러 화면이 되면 안 됨 등 세부 UX 규칙)
4. **안전/톤 QA** — 위기 분기 3중 방어 검증, 공감 전용 원칙 위반 여부(금지어 스캔), 위기 화면에서 게임화 요소 누락 검증
5. **일반 QA** — 상태 전이 커버리지, 엣지 케이스(하루 2회 체크인, 타임존 경계, 429/타임아웃)
